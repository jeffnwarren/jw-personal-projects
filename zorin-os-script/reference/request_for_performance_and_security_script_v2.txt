================================================================================
WHY THIS REWRITE? (META-NOTES ON PROMPTING — for my own learning)
================================================================================

This is a rewrite of request_for_performance_and_security_script.txt. Below are
the prompting principles applied. These notes are for my reference and can be
deleted before submitting to an AI.

1. LEAD WITH CONTEXT, NOT REQUIREMENTS
   The original jumped straight into requirements. The rewrite opens with who
   I am, what the system is, what it's for, and what's already been done. This
   lets the AI calibrate its suggestions to my actual situation rather than
   guessing.

2. SEPARATE FACTS FROM REQUIREMENTS
   The original mixed "here's what I already did" (items 12-13) and "see this
   other file" (item 24) into the numbered requirements list. The rewrite puts
   prior work and reference files in the CONTEXT section and keeps only
   actionable requirements in the REQUIREMENTS section.

3. GROUP BY CONCERN, NOT BY THOUGHT ORDER
   The original listed requirements as they came to mind, so backup, logging,
   error handling, and package management were interleaved. The rewrite groups
   related requirements under clear headings (Performance, Security, Packages,
   Backup, Logging, etc.) so the AI can reason about each domain coherently.

4. USE CONSISTENT, CORRECT TERMINOLOGY
   The original used "security fix" in items 3.a/3.b when the section was
   actually about performance improvements. Small mismatches like this can
   confuse an AI (or a human). The rewrite uses the right term in the right
   place.

5. FIX NUMBERING AND FORMATTING
   The original jumped from item 13 to item 24. Clean, sequential numbering
   and consistent indentation make the document scannable and unambiguous.

6. BE SPECIFIC BUT NOT OVER-PRESCRIPTIVE
   Tell the AI *what* you want (e.g., "backup filenames must include a
   date-time stamp in local timezone"), not *how* to code it (e.g., don't
   write pseudocode for date formatting). The AI is better at implementation
   details than you are — let it do its job.

7. STATE YOUR BOUNDARIES CLEARLY
   "Don't go overboard — this isn't a public-facing server" and "keep the UI
   attractive" are boundary statements that prevent the AI from over-optimizing
   in one direction. The original hinted at this but buried it in sub-items.

8. OWN YOUR DECISIONS; INVITE PUSHBACK ON SPECIFICS
   The original said "I'll likely accept all of your suggestions" which cedes
   too much control and gives the AI no guardrails. The rewrite says "I'm open
   to changes but I want to understand the reasoning." This keeps you in the
   driver's seat while still welcoming the AI's expertise.

9. ASK FOR A PLAN BEFORE CODE
   "Present your proposed scope/plan before writing code so we can align
   first." This one line prevents the AI from writing 500 lines of script
   based on a misunderstanding. Always checkpoint on big tasks.

10. ONE DELIVERABLE, CLEARLY STATED
    The original implied multiple outputs (script, restore script, HANDOFF.md,
    maybe a README) without a clear primary deliverable. The rewrite says
    upfront: "A single downloadable bash script" and then specifies the
    secondary outputs (restore mode, HANDOFF.md) as requirements within it.
